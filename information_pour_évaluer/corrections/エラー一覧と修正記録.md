## 1. 致命的エラー（プログラムがクラッシュ）

### 1.1 リスト初期化の欠如（`damedemain.py`）

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/damedemain.py` — `jeu_possible()` |
| **症状** | `IndexError: list assignment index out of range` |
| **原因** | `J = []` で空リストを作成後、`J[i] = 1` でインデックスアクセスしていた |
| **修正** | `J = [0] * len(diags)` で事前にサイズを確保 |

```python
# 修正前
J = []
J[i] = 1  # IndexError

# 修正後
J = [0] * len(diags)
J[i] = 1  # OK
```

### 1.2 ダム（クイーン）用リスト未初期化（`damedemain.py`）

### 1.3 ユーザー入力のインデックスずれ（`damedemain.py`）

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/damedemain.py` — `tour()` |

| **原因** | ユーザーは1〜8で入力するが、配列は0〜7。変換 `-1` が抜けていた |
| **修正** | `ii = int(input(...)) - 1` |

## 2. ロジックエラー（ゲームが正しく動かない）

### 2.1 敵検出の計算式ミス（`damedemain.py`）— 最重要

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/damedemain.py` — `jeu_possible()` 116〜118行目 |
| **症状** | 駒を飛び越えて捕獲できない（GUIから操作時） |
| **原因** | `(2 - v)` で敵の色を計算していたが、実際には自分の色を返していた |
| **詳細** | `v=0`（白）→ `2-0=2`（白）＝自分自身。`v=1`（黒）→ `2-1=1`（黒）＝自分自身 |
| **修正** | `(1 + v)` に変更。`v=0` → 敵は `1`（黒）、`v=1` → 敵は `2`（白） |

```python
# 修正前（常に自分の駒を敵と誤認）
if L[new_c][new_l][0] == (2 - v):

# 修正後（正しく敵を検出）
if L[new_c][new_l][0] == (1 + v):
```

**影響範囲**: 通常の駒とダムの両方の捕獲ロジックに同一のバグがあり、2箇所修正。

### 2.2 プレイヤー交代の演算ミス（`damedemain.py`）

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/damedemain.py` — `tour()` |
| **原因** | `v += 1 % 2` → 演算子の優先順位で `1 % 2 = 1` → `v += 1` → vが増加し続ける |
| **修正** | `v = (v + 1) % 2` |

### 2.3 条件式の論理ミス（`damedemain.py`）

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/damedemain.py` — `creation_de_jeu()` |
| **原因** | `if fc != 0 or fc != None` は常にTrueになる（ド・モルガンの法則の誤用） |
| **修正** | `if fc != 0 and fc` |

### 2.4 クイーン昇格条件の誤り（`dame_gui_ctk.py`）

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/gui_system/dame_gui_ctk.py` |
| **症状** | 盤面の上下端（行0, 行7）に触れただけでクイーンに昇格 |
| **原因** | 行ベースの判定をしていたが、このゲームでは黒は左から右、白は右から左に進む（列ベース） |
| **修正** | 列ベースに変更: `tc == self.c - 1`（黒の昇格）/ `tc == 0`（白の昇格） |

### 2.5 ダムの移動ロジック崩壊（`damedemain.py`）

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/damedemain.py` — `jeu_possible()` ダム部分 |
| **症状** | ダムが対角線以外のマスに移動できる／捕獲後の着地点が正しくない |
| **原因** | 対角線の走査ロジックが破綻しており、全空マスに移動可と判定していた |
| **修正** | 対角線ごとに `step` を1から増やしながら走査。敵を1つ見つけたら以降は捕獲着地点、味方や2つ目の敵に当たったら停止 |

### 2.6 ダム捕獲時の座標計算ミス（`dame_gui_ctk.py`）

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/gui_system/dame_gui_ctk.py` — `do_move()` |
| **症状** | ダムが敵を飛び越えた時、捕獲された駒の位置がずれる |
| **原因** | `(fc + tc) // 2` で中間点を求めていたが、ダムは複数マス飛べるため中間点≠敵の位置 |
| **修正** | 移動経路を対角線に沿って走査し、実際に駒がある位置を検出して削除 |

### 2.7 通常駒の移動方向制限（`dame_gui_ctk.py`）

| 項目 | 内容 |
|------|------|
| **ファイル** | `src/gui_system/dame_gui_ctk.py` — `get_moves()` |
| **症状** | 通常の駒が捕獲時にも前方にしか進めない |
| **原因** | 捕獲の方向フィルタが通常移動と同じだった |
| **修正** | 捕獲（`J[i] == 1`）は4方向すべてを許可し、通常移動（`J[i] == 2`）のみ前方制限 |

## 3. UX/GUI関連のバグ

### 3.1 ホバーハイライトが消える問題

| 項目 | 内容 |
|------|------|
| **症状** | 駒にカーソルを合わせると移動先が緑に光るが、移動先にカーソルを移動すると即座に消える |
| **原因** | Tkinterのイベント伝播: 子ボタン間を移動する際、親フレームの `<Leave>` イベントが先に発火し、ハイライトをクリアしていた |
| **修正** | `_schedule_clear()` で60msの遅延付きクリアを実装。子ボタンの `<Enter>` が先に発火すれば `_cancel_clear()` でキャンセル |

### 3.2 移動後にハイライトが残る問題

| 項目 | 内容 |
|------|------|
| **症状** | 駒を動かした後も、前回のホバーで緑になったマスが残る |
| **原因** | 移動実行時にハイライトのクリアが移動処理の後に行われていた |
| **修正** | `_execute_move()` 内で `_apply_hover(None, [])` を移動実行前に呼び出し |

### 3.3 パフォーマンス低下（GUI遅延）

| 項目 | 内容 |
|------|------|
| **症状** | マウスを動かすとGUIが重くなる |
| **原因** | 毎回の `<Enter>` イベントで全64マスを再描画していた |
| **修正** | 差分のみ更新する `_apply_hover()` を実装。変更のあったセルのみ `fg_color` を更新 |
